Unit I: Java Fundamentals

1. Define JDK, JRE, and JVM. Explain their relationship.

JVM (Java Virtual Machine):

  • Abstract machine that provides runtime environment to execute Java bytecode
  • Platform-dependent (different JVM for Windows, Linux, Mac)
  • Performs memory management, garbage collection, security
  • Components: Class Loader, Bytecode Verifier, Execution Engine

JRE (Java Runtime Environment):

  • JVM + Java class libraries + supporting files
  • Provides runtime environment to run Java applications
  • Cannot compile Java programs (no development tools)
  • End users need JRE to run Java applications

JDK (Java Development Kit):

  • JRE + Development tools (javac compiler, debugger, javadoc)
  • Required by developers to write and compile Java programs
  • Includes tools like jar, jdb, javap

Relationship: JDK ⊃ JRE ⊃ JVM

JDK = JRE + Development Tools
JRE = JVM + Library Classes
JVM = Runtime Engine
2. Why is Java called platform-independent? Explain with diagram.

Java achieves platform independence through its "Write Once, Run Anywhere" (WORA) philosophy:

How it works:

  1. Java source code (.java) is compiled by javac compiler
  2. Compiler produces bytecode (.class files), not machine code
  3. Bytecode is platform-independent intermediate code
  4. JVM on any platform interprets/executes this bytecode
  5. JVM converts bytecode to machine-specific instructions 
Source Code (.java)
        ↓
   [javac compiler]
        ↓
  Bytecode (.class)  ← Platform Independent
        ↓
   [JVM - Windows]  [JVM - Linux]  [JVM - Mac]
        ↓               ↓              ↓
   Windows Code    Linux Code     Mac Code

Key Points:

  • Java is platform-independent, but JVM is platform-dependent
  • Same bytecode runs on any OS with compatible JVM
  • This differs from C/C++ which compiles directly to machine code
3. Explain the four pillars of OOP with examples.

1. Encapsulation:

Bundling data (variables) and methods that operate on data into a single unit (class), hiding internal state. 

class BankAccount {
    private double balance;  // Hidden data

    public void deposit(double amount) {  // Controlled access
        if (amount > 0) balance += amount;
    }

    public double getBalance() {
        return balance;
    }
}

2. Inheritance:

Creating new classes from existing ones, inheriting properties and behaviors. Promotes code reuse. 

class Animal {
    void eat() { System.out.println("Eating..."); }
}

class Dog extends Animal {  // Dog inherits from Animal
    void bark() { System.out.println("Barking..."); }
}

// Dog can use eat() from Animal

3. Polymorphism:

One interface, multiple implementations. Same method behaves differently based on object. 

// Compile-time (Overloading)
void print(int x) { }
void print(String s) { }

// Runtime (Overriding)
class Shape { void draw() { } }
class Circle extends Shape {
    void draw() { System.out.println("Drawing Circle"); }
}
class Rectangle extends Shape {
    void draw() { System.out.println("Drawing Rectangle"); }
}

Shape s = new Circle();
s.draw();  // "Drawing Circle" - decided at runtime

4. Abstraction:

Hiding complex implementation details, showing only essential features. 

abstract class Vehicle {
    abstract void start();  // What to do, not how
}

class Car extends Vehicle {
    void start() {
        // Complex implementation hidden
        System.out.println("Car started");
    }
}

7. Explain == vs equals() with examples.

== Operator:

  • Compares references (memory addresses) for objects
  • Compares actual values for primitives
  • Returns true if both references point to same object

equals() Method:

  • Defined in Object class, can be overridden
  • Default implementation uses == (reference comparison)
  • String and wrapper classes override it for content comparison 
// Example with Strings
String s1 = new String("Hello");
String s2 = new String("Hello");
String s3 = "Hello";
String s4 = "Hello";

System.out.println(s1 == s2);        // false (different objects)
System.out.println(s1.equals(s2));   // true (same content)
System.out.println(s3 == s4);        // true (String pool - same reference)
System.out.println(s3.equals(s4));   // true (same content)

// Example with primitives
int a = 5, b = 5;
System.out.println(a == b);          // true (value comparison)

// Example with custom class
class Person {
    String name;
    Person(String name) { this.name = name; }

    @Override
    public boolean equals(Object obj) {
        if (obj instanceof Person) {
            return this.name.equals(((Person) obj).name);
        }
        return false;
    }
}

Person p1 = new Person("John");
Person p2 = new Person("John");
System.out.println(p1 == p2);        // false
System.out.println(p1.equals(p2));   // true (overridden)
8. Explain String pool and String immutability.

String Pool (String Intern Pool):

  • Special memory area in heap for storing String literals
  • When String literal is created, JVM checks pool first
  • If exists, returns existing reference; else creates new entry
  • Saves memory by reusing identical strings
String s1 = "Hello";  // Creates in pool
String s2 = "Hello";  // Returns same reference from pool
String s3 = new String("Hello");  // Creates new object in heap

System.out.println(s1 == s2);  // true (same pool reference)
System.out.println(s1 == s3);  // false (different objects)

// intern() moves string to pool
String s4 = s3.intern();
System.out.println(s1 == s4);  // true

String Immutability:

  • Once created, String content cannot be changed
  • Any modification creates a new String object
  • String class is final (cannot be extended)

Why Strings are Immutable:

  1. Security: Strings used in network connections, database URLs, file paths
  2. String Pool: Multiple references can safely share same String
  3. Thread Safety: Immutable objects are inherently thread-safe
  4. Hashcode Caching: Hash can be cached for HashMap keys 
String s = "Hello";
s.concat(" World");  // Creates new String, s unchanged
System.out.println(s);  // "Hello"

s = s.concat(" World");  // s now points to new String
System.out.println(s);  // "Hello World"

10. What is a constructor? Explain types and constructor overloading.

Constructor:

  • Special method called automatically when object is created
  • Same name as class, no return type (not even void)
  • Used to initialize object state
  • Cannot be static, final, or abstract

Types of Constructors:

1. Default Constructor:

class Student {
    String name;
    // Compiler provides: Student() { }
}
Student s = new Student();  // Uses default constructor

2. Parameterized Constructor:

class Student {
    String name;
    int age;

    Student(String name, int age) {
        this.name = name;
        this.age = age;
    }
}
Student s = new Student("John", 20);

3. Copy Constructor (User-defined): 

class Student {
    String name;

    Student(Student other) {
        this.name = other.name;
    }
}
Student s1 = new Student("John");
Student s2 = new Student(s1);  // Copy of s1

Constructor Overloading:

class Rectangle {
    int length, width;

    Rectangle() {  // Default
        length = width = 1;
    }

    Rectangle(int side) {  // Square
        length = width = side;
    }

    Rectangle(int l, int w) {  // Custom
        length = l;
        width = w;
    }
}

Rectangle r1 = new Rectangle();        // 1x1
Rectangle r2 = new Rectangle(5);       // 5x5
Rectangle r3 = new Rectangle(4, 6);    // 4x6
11. Explain this and super keywords with examples.

this Keyword:

  • Refers to current object instance
  • Used to distinguish instance variables from parameters
  • Can call current class constructor: this()
  • Can pass current object as argument
class Student {
    String name;
    int age;

    Student(String name, int age) {
        this.name = name;  // this.name = instance variable
        this.age = age;
    }

    Student(String name) {
        this(name, 18);  // Calls parameterized constructor
    }

    void display() {
        print(this);  // Pass current object
    }
}

super Keyword:

  • Refers to immediate parent class object
  • Used to access parent class members (hidden by child)
  • Can call parent class constructor: super()
  • Must be first statement in constructor
class Animal {
    String name = "Animal";

    Animal(String name) {
        this.name = name;
    }

    void display() {
        System.out.println("Animal display");
    }
}

class Dog extends Animal {
    String name = "Dog";

    Dog(String name) {
        super(name);  // Call parent constructor
    }

    void display() {
        System.out.println(name);        // Dog
        System.out.println(super.name);  // Animal
        super.display();  // Call parent method
    }
}
12. Explain static keyword - variables, methods, blocks, and nested classes.

Static members belong to class, not instances:

1. Static Variables:

  • One copy shared by all instances
  • Memory allocated when class loads
  • Also called class variables
class Counter {
    static int count = 0;  // Shared
    int id;                // Unique per object

    Counter() {
        count++;
        id = count;
    }
}
Counter c1 = new Counter();  // count=1, c1.id=1
Counter c2 = new Counter();  // count=2, c2.id=2
System.out.println(Counter.count);  // 2

2. Static Methods:

  • Called without creating object
  • Cannot access instance variables/methods directly
  • Cannot use this or super
class MathUtils {
    static int square(int n) {
        return n * n;
    }
}
int result = MathUtils.square(5);  // 25

3. Static Blocks:

  • Executed once when class is loaded
  • Used for static initialization
class Database {
    static Connection conn;

    static {
        System.out.println("Class loading...");
        conn = createConnection();
    }
}

4. Static Nested Classes:

class Outer {
    static class Nested {
        void display() {
            System.out.println("Static nested class");
        }
    }
}
Outer.Nested obj = new Outer.Nested();
13. Explain final keyword with all use cases.

1. Final Variable (Constant):

  • Value cannot be changed once assigned
  • Must be initialized (at declaration, constructor, or static block)
  • Convention: UPPER_CASE names
class Circle {
    final double PI = 3.14159;       // Initialized at declaration
    final int radius;

    Circle(int r) {
        radius = r;                   // Initialized in constructor
    }
}

final int MAX_VALUE = 100;
MAX_VALUE = 200;  // ERROR: cannot reassign

2. Final Method:

  • Cannot be overridden by subclasses
  • Used for security and design constraints
class Parent {
    final void display() {
        System.out.println("Cannot override this");
    }
}

class Child extends Parent {
    // void display() { }  // ERROR: cannot override final method
}

3. Final Class:

  • Cannot be extended (no inheritance)
  • Example: String, Integer, Math classes
final class ImmutableClass {
    // Class content
}

// class ExtendedClass extends ImmutableClass { }  // ERROR

4. Final Parameters:

void process(final int value) {
    // value = 10;  // ERROR: cannot modify final parameter
}

Final vs Immutability:

final List<String> list = new ArrayList<>();
list.add("Hello");    // OK - modifying contents
// list = new ArrayList<>();  // ERROR - reassigning reference

15. Explain garbage collection and memory management in Java.

Garbage Collection:

  • Automatic memory management by JVM
  • Reclaims memory from objects no longer in use
  • Runs in background as a daemon thread
  • Cannot be forced, only requested via System.gc()

When an Object Becomes Eligible for GC:

  1. Reference set to null: obj = null;
  2. Reference reassigned: obj = new Object();
  3. Object created inside method (after method returns)
  4. Island of isolation (objects referencing each other, but unreachable)

Memory Areas:

┌─────────────────────────────┐
│          Method Area        │ ← Class metadata, static vars
├─────────────────────────────┤
│            Heap             │ ← Objects, instance variables
│  ┌────────┬───────────────┐ │
│  │ Young  │     Old       │ │
│  │  Gen   │  Generation   │ │
│  └────────┴───────────────┘ │
├─────────────────────────────┤
│     Stack (per thread)      │ ← Local variables, method calls
├─────────────────────────────┤
│     PC Registers            │ ← Current instruction
├─────────────────────────────┤
│     Native Method Stack     │ ← Native method info
└─────────────────────────────┘

finalize() Method:

class Resource {
    @Override
    protected void finalize() throws Throwable {
        System.out.println("Cleanup before GC");
        // Release resources
        super.finalize();
    }
}

Best Practices:

  • Nullify references when done with large objects
  • Use try-with-resources for closeable resources
  • Avoid creating unnecessary objects in loops
  • Use StringBuilder for string concatenation
16. Why is multiple inheritance not supported in Java? How to achieve it?

The Diamond Problem:

If Java allowed multiple inheritance with classes: 

class A {
    void display() { System.out.println("A"); }
}
class B extends A {
    void display() { System.out.println("B"); }
}
class C extends A {
    void display() { System.out.println("C"); }
}
class D extends B, C { }  // NOT ALLOWED!
// Which display() would D inherit? Ambiguity!

Solution: Interfaces

Java supports multiple inheritance through interfaces: 

interface Printable {
    void print();
}

interface Showable {
    void show();
}

class Document implements Printable, Showable {
    public void print() {
        System.out.println("Printing document");
    }

    public void show() {
        System.out.println("Showing document");
    }
}

Default Methods (Java 8+) Conflict: 

interface A {
    default void display() { System.out.println("A"); }
}
interface B {
    default void display() { System.out.println("B"); }
}

class C implements A, B {
    // Must override to resolve conflict
    public void display() {
        A.super.display();  // Choose A's implementation
        // or B.super.display();
        // or provide own implementation
    }
}

16. What is the difference between String, StringBuilder, and StringBuffer?

String: Immutable. Every modification creates a new object. Thread-safe by immutability.

StringBuilder: Mutable. More efficient for concatenation. Not thread-safe.

StringBuffer: Mutable and synchronized. Thread-safe but slower than StringBuilder.

Use case: String for fixed values, StringBuilder for single-threaded string manipulation, StringBuffer for multi-threaded scenarios.

17. What is method overloading and what are its rules?

Method overloading: Same method name, different parameters in same class. Compile-time polymorphism.

Rules:

  • Must have different parameter list (number or types)
  • Return type can be different but doesn't contribute to overloading
  • Access modifiers can be different
  • Can throw different exceptions
18. What is method overriding and what are its rules?

Method overriding: Subclass provides specific implementation of parent class method. Runtime polymorphism.

Rules:

  • Must have same signature as parent method
  • Return type must be same or covariant
  • Access modifier cannot be more restrictive
  • Cannot override static, final, or private methods
  • @Override annotation recommended
19. What is the difference between == and equals() method?

== operator: Compares references (memory addresses) for objects. For primitives, compares values.

equals() method: Compares content/values of objects. Defined in Object class, can be overridden.

Example: String s1 = "Hello"; String s2 = new String("Hello");
s1 == s2 returns false (different objects)
s1.equals(s2) returns true (same content)

26. Explain interfaces in Java with a suitable example.

An interface is a pure abstract type that defines a contract — a set of method signatures that any implementing class must fulfil. It enables 100% abstraction and is Java's mechanism for achieving multiple inheritance. Since Java 8, interfaces can also have default and static methods with bodies.

Key Characteristics:

  • All methods are public abstract by default (before Java 8)
  • All variables are public static final (constants) by default
  • A class can implement multiple interfaces (multiple inheritance)
  • An interface can extend multiple interfaces
  • Cannot be instantiated directly
// ── Defining an Interface ──────────────────────────────────────
interface Shape {
    double PI = 3.14159;       // implicitly public static final

    double area();             // implicitly public abstract
    double perimeter();

    // Default method (Java 8+) — optional override
    default void displayInfo() {
        System.out.println("Area: " + area());
        System.out.println("Perimeter: " + perimeter());
    }

    // Static method (Java 8+) — called on interface name

27. Explain the concept of autoboxing and unboxing with examples.

Java's primitive types (int, double, etc.) are not objects, so they cannot be stored in collections or used with generics. Autoboxing is the automatic conversion of a primitive to its corresponding wrapper class object. Unboxing is the reverse — automatic conversion of a wrapper object back to a primitive. Both happen transparently at compile time. 

// ── Autoboxing: primitive → Wrapper object ──────────────────────
int  primitiveInt = 42;
Integer boxedInt  = primitiveInt;     // Autoboxing: int → Integer

double primitiveDouble = 3.14;
Double boxedDouble     = primitiveDouble;  // double → Double

boolean primBool = true;
Boolean boxedBool = primBool;         // boolean → Boolean

// ── Unboxing: Wrapper object → primitive ────────────────────────
Integer obj = Integer.valueOf(100);
int num = obj;                        // Unboxing: Integer → int

Double dObj = 9.8;
double d = dObj;                      // Unboxing: Double → double

// ── In Collections (autoboxing is essential) ────────────────────
List<Integer> list = new ArrayList<>();
list.add(10);   // Autoboxing: int 10 → Integer.valueOf(10)
list.add(20);

int sum = 0;
for (int val : list) {  // Unboxing: Integer → int each iteration
    sum += val;
}
System.out.println(sum);  // 30

// ── Arithmetic triggers unboxing ────────────────────────────────
Integer a = 50;
Integer b = 30;
int result = a + b;       // a and b are unboxed before addition

// ── Integer Cache (-128 to 127) ─────────────────────────────────
Integer x = 127;
Integer y = 127;
System.out.println(x == y);       // true  (same cached object)

Integer p = 200;
Integer q = 200;
System.out.println(p == q);       // false (new objects, > 127)
System.out.println(p.equals(q));  // true  (always use equals!)

// ── NullPointerException trap ───────────────────────────────────
Integer nullInt = null;
int value = nullInt;  // NullPointerException during unboxing!

Autoboxing vs Manual Boxing:

Action Manual (old style) Autoboxing (modern)
Box Integer.valueOf(5) Integer x = 5;
Unbox x.intValue() int n = x;
Parse String Integer.parseInt("5") No shortcut — still manual

Key Points:

  • Autoboxing is a compile-time syntactic sugar — the compiler inserts Integer.valueOf() and intValue() calls automatically
  • Avoid autoboxing in performance-critical loops — repeated boxing/unboxing creates garbage objects
  • Always use .equals() to compare wrapper objects, not ==
  • Unboxing a null wrapper throws NullPointerException

5. What is try-with-resources? Explain with example.

Automatic resource management introduced in Java 7. Resources are automatically closed after try block.

Requirements:

  • Resource must implement AutoCloseable interface
  • Resources declared in try parentheses
  • Multiple resources separated by semicolons
  • Resources closed in reverse order of declaration 
// Traditional way (verbose)

try {
    br = new BufferedReader(new FileReader("file.txt"));
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try {
            br.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// Try-with-resources (clean)
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
// br is automatically closed here

// Multiple resources
try (FileInputStream fis = new FileInputStream("input.txt");
     FileOutputStream fos = new FileOutputStream("output.txt")) {
    // Copy file
    int data;
    while ((data = fis.read()) != -1) {
        fos.write(data);
    }
}  // Both streams auto-closed (fos closed first, then fis)

13. Explain wait(), notify(), and notifyAll() methods.

Inter-thread Communication Methods (Object class):

  • wait(): Releases lock and waits until notified
  • notify(): Wakes up one waiting thread
  • notifyAll(): Wakes up all waiting threads

Rules:

  • Must be called from synchronized context
  • Must own the object's monitor (lock)
  • IllegalMonitorStateException if rules violated 
class SharedResource {
    private int data;
    private boolean hasData = false;

    public synchronized void produce(int value) throws InterruptedException {
        while (hasData) {
            wait();  // Wait until consumed
        }
        data = value;
        System.out.println("Produced: " + value);
        hasData = true;
        notify();  // Notify consumer
    }

    public synchronized int consume() throws InterruptedException {
        while (!hasData) {
            wait();  // Wait until produced
        }
        hasData = false;
        notify();  // Notify producer
        System.out.println("Consumed: " + data);
        return data;
    }
}

14. What is deadlock? How to prevent it?

Deadlock: Situation where two or more threads are blocked forever, each waiting for the other's lock.

Conditions for Deadlock:

  1. Mutual Exclusion: Resource can only be held by one thread
  2. Hold and Wait: Thread holds resource while waiting for another
  3. No Preemption: Resources cannot be forcibly taken
  4. Circular Wait: Circular chain of threads waiting for resources

Deadlock Example:

Object lock1 = new Object();
Object lock2 = new Object();

// Thread 1
new Thread(() -> {
    synchronized (lock1) {
        System.out.println("Thread 1: Holding lock1");
        try { Thread.sleep(100); } catch (Exception e) {}

        synchronized (lock2) {  // Waiting for lock2
            System.out.println("Thread 1: Holding lock1 and lock2");
        }
    }
}).start();

// Thread 2
new Thread(() -> {
    synchronized (lock2) {
        System.out.println("Thread 2: Holding lock2");
        try { Thread.sleep(100); } catch (Exception e) {}

        synchronized (lock1) {  // Waiting for lock1 - DEADLOCK!
            System.out.println("Thread 2: Holding lock2 and lock1");
        }
    }
}).start();

Prevention Strategies:

  1. Lock Ordering: Always acquire locks in same order 
    // Thread 1 and Thread 2 both do:
synchronized (lock1) {
    synchronized (lock2) {
        // No deadlock
    }
}
  2. Lock Timeout: Use tryLock() with timeout 
    if (lock.tryLock(1, TimeUnit.SECONDS)) {
    try { /* work */ } finally { lock.unlock(); }
}
  3. Avoid Nested Locks: Minimize synchronized blocks
  4. Use Higher-level Concurrency: java.util.concurrent utilities

21. What is the purpose of the finally block in Java exception handling?

The finally block is an optional block that always executes after a try-catch sequence, regardless of whether an exception occurred, was caught, or was not caught. Its primary purpose is to perform cleanup actions — releasing resources, closing connections, or unlocking locks — guaranteeing they run even when exceptions disrupt normal flow.

When finally ALWAYS runs:

  • Normal execution (no exception thrown)
  • Exception thrown and caught by a catch block
  • Exception thrown but NOT caught (finally runs before propagation)
  • When a return statement is inside try or catch

When finally does NOT run:

  • System.exit() is called inside try or catch
  • JVM crashes (e.g., power failure, fatal error)
  • The thread is forcefully killed
// ── Scenario 1: Normal execution ─────────────────────────────
try {
    System.out.println("try block");
} catch (Exception e) {
    System.out.println("catch block");
} finally {
    System.out.println("finally block");  // Runs
}
// Output: try block  →  finally block

// ── Scenario 2: Exception caught ───────────────────────────
try {
    int result = 10 / 0;  // ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("caught: " + e.getMessage());
} finally {
    System.out.println("finally runs");   // Runs
}

// ── Scenario 3: Exception NOT caught ────────────────────────
try {
    String s = null;
    s.length();           // NullPointerException
} finally {
    System.out.println("finally runs");   // Runs before propagation
}  // NullPointerException propagates to caller

// ── Scenario 4: return in try ──────────────────────────────
public int computeValue() {
    try {
        return 100;                       // Will return 100
    } finally {
        System.out.println("cleanup!");   // Runs BEFORE the return
        // return 200;  // BAD PRACTICE: overrides try's return value!
    }
}  // Returns 100, but "cleanup!" is printed first

// ── Classic use-case: resource cleanup ─────────────────────────
Connection conn = null;
try {
    conn = DriverManager.getConnection(url, user, pass);
    // ... execute queries
} catch (SQLException e) {
    System.err.println("DB error: " + e.getMessage());
} finally {
    if (conn != null) {
        try { conn.close(); }     // Always close connection
        catch (SQLException e) { e.printStackTrace(); }
    }
}
// Note: Java 7+ try-with-resources is the modern preferred alternative

Key Points:

  • Guaranteed execution — the cornerstone of reliable resource management
  • A return in finally overrides any return in try/catch — avoid this pattern
  • An exception thrown in finally suppresses the original exception — always use try-with-resources for auto-closeable resources instead
  • Can be used without a catch block: try { } finally { } is valid
22. Write a Java program where two threads print even and odd numbers separately and alternatively using wait() and notify().

This classic inter-thread communication problem uses a shared monitor object along with wait() and notify() to alternate execution between two threads. One thread prints odd numbers; the other prints even numbers, producing a perfectly sequential output: 1, 2, 3, 4, 5, ... 

// ──────────────────────────────────────────────────────────────────────
public class EvenOddPrinter {

    private static final Object LOCK = new Object();  // Shared monitor
    private static int number = 1;                    // Shared counter
    private static final int MAX = 10;                // Print up to 10

    // Thread that prints ODD numbers (1, 3, 5 ...)
    static class OddPrinter implements Runnable {
        @Override
        public void run() {
            while (number <= MAX) {
                synchronized (LOCK) {
                    // If current number is EVEN, wait for EvenPrinter
                    while (number <= MAX && number % 2 == 0) {
                        try { LOCK.wait(); }
                        catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                    if (number <= MAX) {
                        System.out.println("ODD  Thread: " + number);
                        number++;            // Advance to next number
                        LOCK.notify();       // Wake up EvenPrinter
                    }
                }
            }
        }
    }

    // Thread that prints EVEN numbers (2, 4, 6 ...)
    static class EvenPrinter implements Runnable {
        @Override
        public void run() {
            while (number <= MAX) {
                synchronized (LOCK) {
                    // If current number is ODD, wait for OddPrinter
                    while (number <= MAX && number % 2 != 0) {
                        try { LOCK.wait(); }
                        catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }
                    if (number <= MAX) {
                        System.out.println("EVEN Thread: " + number);
                        number++;            // Advance to next number
                        LOCK.notify();       // Wake up OddPrinter
                    }
                }
            }
        }
    }

    public static void main(String[] args) throws InterruptedException {
        Thread oddThread  = new Thread(new OddPrinter(),  "OddThread");
        Thread evenThread = new Thread(new EvenPrinter(), "EvenThread");

        oddThread.start();    // Start odd thread first (number starts at 1)
        evenThread.start();

        oddThread.join();
        evenThread.join();
        System.out.println("Done!");
    }
}

// Expected Output:
// ODD  Thread: 1
// EVEN Thread: 2
// ODD  Thread: 3
// EVEN Thread: 4
// ODD  Thread: 5
// EVEN Thread: 6
// ODD  Thread: 7
// EVEN Thread: 8
// ODD  Thread: 9
// EVEN Thread: 10
// Done!

How it works:

  • Both threads share a single LOCK object and the number variable
  • OddPrinter: enters synchronized(LOCK), checks if number is even — if so, calls LOCK.wait() (releases lock and sleeps). When notified, it checks again, prints the odd number, increments, and calls notify()
  • EvenPrinter: does the mirror image — waits if number is odd, prints even, increments, notifies
  • The while (condition) { wait(); } pattern (instead of if) guards against spurious wakeups

24. Explain transient and volatile keywords in Java.

Both transient and volatile are special Java field modifiers, but they serve completely different purposes: transient controls serialization, while volatile controls visibility in multithreaded programs.

transient keyword — Serialization control:

A field marked transient is excluded from the serialization process. When the object is serialized to a byte stream, transient fields are skipped and restored to their default value upon deserialization. 

import java.io.*;

class UserAccount implements Serializable {
    private static final long serialVersionUID = 1L;

    private String username;            // Will be serialized
    private String email;               // Will be serialized
    private transient String password;  // EXCLUDED from serialization
    private transient int loginCount;   // EXCLUDED (no need to persist)
    private transient java.sql.Connection conn;  // Not serializable itself

    public UserAccount(String username, String email, String password) {
        this.username  = username;
        this.email     = email;
        this.password  = password;
        this.loginCount = 0;
    }

    @Override
    public String toString() {
        return "User{user='" + username + "', email='" + email +
               "', password='" + password + "', loginCount=" + loginCount + "}";
    }
}

// Serialization
UserAccount acc = new UserAccount("john", "john@x.com", "secret123");
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("user.ser"))) {
    oos.writeObject(acc);
}

// Deserialization
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("user.ser"))) {
    UserAccount loaded = (UserAccount) ois.readObject();
    System.out.println(loaded);
    // User{user='john', email='john@x.com', password='null', loginCount=0}
    // password=null, loginCount=0 — transient fields reset to defaults!
}

volatile keyword — Thread visibility:

A field marked volatile tells the JVM to always read from and write to main memory, never from a thread's local CPU cache. This guarantees visibility of changes across threads but does not guarantee atomicity. 

// ── Problem WITHOUT volatile ──────────────────────────────────────
class StopFlag {
    private boolean running = true;  // NOT volatile

    public void stop() { running = false; }

    public void doWork() {
        while (running) {            // Thread may cache 'running = true'
            // ...work...            // and NEVER see the change from stop()
        }
    }
}

// ── Fixed WITH volatile ────────────────────────────────────────────
class SafeStopFlag {
    private volatile boolean running = true; // Always reads from main memory

    public void stop() { running = false; }  // Write is immediately visible

    public void doWork() {
        while (running) {            // Always reads latest value
            // ...work...
        }
        System.out.println("Stopped cleanly.");
    }
}

// ── volatile does NOT guarantee atomicity ──────────────────────────
volatile int counter = 0;
counter++;  // NOT atomic! Read + increment + write = 3 operations
            // Use AtomicInteger instead:
AtomicInteger atomicCounter = new AtomicInteger(0);
atomicCounter.incrementAndGet();  // Thread-safe atomic increment

transient vs volatile — Side-by-side:

Aspect transient volatile
Purpose Exclude field from serialization Ensure multi-thread visibility of field
Context Serialization / persistence Multithreading / concurrency
Effect on value Field becomes default (null/0) after deserializing Field is read from main memory on every access
Thread safety? Not related Visibility only — not atomicity
Can combine? Yes: private transient volatile int x;

Key Points:

  • Use transient for sensitive data (passwords), derived/cached fields, or fields holding non-serializable objects
  • Use volatile for simple flags shared between threads where only one thread writes and many threads read
  • Neither transient nor volatile applies to local variables — only instance/class fields

7. Explain the internal working of HashMap.

Structure: Array of buckets, each bucket is linked list (or tree for Java 8+). 

// Internal structure (simplified)
class HashMap<K, V> {
    Node<K,V>[] table;  // Array of buckets
    int size;
    float loadFactor;   // Default 0.75
    int threshold;      // capacity * loadFactor

    static class Node<K,V> {
        int hash;
        K key;
        V value;
        Node<K,V> next;
    }
}

put() Operation:

  1. Calculate hash: hash = key.hashCode() ^ (hash >>> 16)
  2. Find bucket index: index = hash & (capacity - 1)
  3. If bucket empty, create new node
  4. If key exists (equals check), update value
  5. Else, add to linked list (end) or tree
  6. If size > threshold, resize (double capacity)

get() Operation:

  1. Calculate hash and find bucket
  2. Search bucket (linked list or tree) using equals()
  3. Return value if found, null otherwise
// Hash collision handling
// Java 8+: When bucket has > 8 nodes, converts to tree (O(log n))
// When nodes < 6, converts back to linked list

// Rehashing when capacity doubles
// All entries are re-hashed to new buckets

Performance:

  • Best case: O(1) - no collisions
  • Worst case (before Java 8): O(n) - all in one bucket
  • Worst case (Java 8+): O(log n) - tree bucket
8. Explain hashCode() and equals() contract.

The Contract:

  1. If a.equals(b) == true, then a.hashCode() == b.hashCode()
  2. If a.hashCode() != b.hashCode(), then a.equals(b) == false
  3. If a.hashCode() == b.hashCode(), a.equals(b) may or may not be true (collision)
  4. hashCode() must return same value for same object across invocations 
class Person {
    private String name;
    private int age;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Person person = (Person) o;
        return age == person.age && Objects.equals(name, person.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}

// Problems without proper implementation
Map<Person, String> map = new HashMap<>();
Person p1 = new Person("John", 25);
map.put(p1, "Engineer");

Person p2 = new Person("John", 25);  // Same data as p1

                           // "Engineer" if properly overridden

Why Override Both:

  • HashMap uses hashCode() to find bucket
  • Then uses equals() to find exact match in bucket
  • If only equals() overridden: keys with same data go to different buckets
  • If only hashCode() overridden: same hash but never equal

16. How does Stack differ from Queue? Explain with code examples for both Stack and Queue operations.

Stack follows LIFO (Last-In-First-Out) order — the last element inserted is the first to be removed, like a pile of plates. Queue follows FIFO (First-In-First-Out) order — the first element inserted is the first to be removed, like a queue at a ticket counter.

Aspect Stack (LIFO) Queue (FIFO)
Order Last-In-First-Out First-In-First-Out
Insert push() — at the top offer()/add() — at the rear/tail
Remove pop() — from the top poll()/remove() — from the front/head
Peek peek() — view top without removing peek() — view front without removing
Java Class Stack<T> (legacy), Deque preferred Queue<T>, LinkedList, ArrayDeque
Use cases Undo/redo, recursion, expression parsing Print queue, task scheduling, BFS 
import java.util.*;

public class StackVsQueue {
    public static void main(String[] args) {

        // ── STACK ── Using Deque (recommended over legacy Stack class)
        Deque<Integer> stack = new ArrayDeque<>();

        stack.push(10);           // Push: adds to top
        stack.push(20);
        stack.push(30);
        System.out.println("Stack after pushes: " + stack); // [30, 20, 10]

        System.out.println("peek(): " + stack.peek());       // 30 (top, not removed)
        System.out.println("pop():  " + stack.pop());        // 30 (removed)
        System.out.println("pop():  " + stack.pop());        // 20
        System.out.println("Stack after pops:   " + stack);  // [10]

        // ── QUEUE ── Using LinkedList / ArrayDeque
        Queue<Integer> queue = new LinkedList<>();

        queue.offer(10);          // Offer: adds to rear
        queue.offer(20);
        queue.offer(30);
        System.out.println("Queue after offers: " + queue); // [10, 20, 30]

        System.out.println("peek(): " + queue.peek());       // 10 (front, not removed)
        System.out.println("poll(): " + queue.poll());       // 10 (removed)
        System.out.println("poll(): " + queue.poll());       // 20
        System.out.println("Queue after polls:  " + queue);  // [30]

        // ── Legacy Stack class (still works but Deque is preferred)
        Stack<String> legacyStack = new Stack<>();
        legacyStack.push("A");
        legacyStack.push("B");
        legacyStack.push("C");
        System.out.println("Top: " + legacyStack.peek());   // C
        System.out.println("Pop: " + legacyStack.pop());    // C
        System.out.println("isEmpty: " + legacyStack.isEmpty()); // false

        // ── Real-world use: Stack for balanced bracket checking
        String expr = "({[]})"
        Deque<Character> checkStack = new ArrayDeque<>();
        for (char c : expr.toCharArray()) {
            if (c == '(' || c == '{' || c == '[') {
                checkStack.push(c);
            } else {
                if (checkStack.isEmpty()) break;
                char top = checkStack.pop();
                if ((c==')' && top!='(') || (c=='}' && top!='{') || (c==']' && top!='[')) {
                    System.out.println("Unbalanced!");
                }
            }
        }
        System.out.println(checkStack.isEmpty() ? "Balanced!" : "Unbalanced!");
        // Output: Balanced!
    }
}

Key Points:

  • Prefer ArrayDeque over the legacy Stack class (faster, no synchronization overhead)
  • Use offer()/poll() for Queue operations instead of add()/remove() to avoid exceptions on empty collections
  • PriorityQueue is a special queue where elements are dequeued in priority (natural or custom) order — not strict FIFO
17. What is the difference between Iterator and for-each loop?

Both Iterator and the enhanced for-each loop are used to traverse collections, but they differ in flexibility, control, and use cases. The for-each loop is syntactic sugar — the compiler converts it into an Iterator behind the scenes.

Aspect Iterator for-each (Enhanced for)
Syntax Verbose (explicit iterator) Concise and readable
Remove elements? Yes — iterator.remove() No — ConcurrentModificationException
Control over traversal Full control (skip, conditional) No control — traverses all elements
Works on Any Iterable / Collection Any Iterable, arrays
Access to index? No (use ListIterator) No
Null collection? Throws NullPointerException Throws NullPointerException
Performance Same (for-each compiles to iterator) Same 
import java.util.*;

List<String> names = new ArrayList<>(Arrays.asList("Alice", "Bob", "Charlie", "Dave"));

// ── for-each loop ─────────────────────────────────────────────────────────
for (String name : names) {
    System.out.println(name);     // Simple and readable
    // names.remove(name);        // ConcurrentModificationException!
}

// ── Iterator: safe removal during traversal ────────────────────────────
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    String name = it.next();
    if (name.startsWith("B")) {
        it.remove();              // Safe removal
    }
}
System.out.println(names);        // [Alice, Charlie, Dave]

// ── What the compiler generates for a for-each loop ──────────────────────
// for (String n : names) { ... }
// ↓↓ is exactly equivalent to:
Iterator<String> _it = names.iterator();
while (_it.hasNext()) {
    String n = _it.next();
    // ... loop body
}

// ── for-each on arrays ────────────────────────────────────────────────────────
int[] arr = {1, 2, 3, 4, 5};
for (int n : arr) {              // Works on arrays too (no Iterator needed)
    System.out.print(n + " ");   // 1 2 3 4 5
}

// ── Iterator advantage: conditional skip ────────────────────────────────────
Iterator<Integer> numIt = numbers.iterator();
while (numIt.hasNext()) {
    int num = numIt.next();
    if (num % 2 == 0) continue;  // Skip evens (same as for-each)
    if (num > 7) break;          // Stop early (same as for-each)
    System.out.print(num + " ");  // 1 3 5 7
}

Key Points:

  • Use for-each for simple read-only traversal — cleaner and less error-prone
  • Use Iterator when you need to remove elements during traversal or need conditional/early-exit control with removal
  • For-each cannot modify the collection structure; attempting to do so throws ConcurrentModificationException
  • Use removeIf() (Java 8+) as a clean alternative to Iterator-based removal: names.removeIf(n -> n.startsWith("B"));
18. What is the difference between fail-fast and fail-safe iterators?

When a collection is modified (add/remove) while being iterated, a fail-fast iterator immediately throws ConcurrentModificationException, making the failure visible right away. A fail-safe iterator works on a snapshot or copy of the collection and does not throw an exception, but may not reflect the latest changes.

Aspect Fail-Fast Iterator Fail-Safe Iterator
Throws exception? Yes — ConcurrentModificationException No — continues safely
Works on Original collection Copy / snapshot of collection
Sees new changes? N/A (throws on change) No — iterates over snapshot
Memory overhead Low (no copy) Higher (creates copy)
Examples ArrayList, HashMap, HashSet iterators CopyOnWriteArrayList, ConcurrentHashMap
Thread safety Not thread-safe Thread-safe 
import java.util.*;
import java.util.concurrent.*;

// ── FAIL-FAST: ArrayList iterator ────────────────────────────────────────
List<String> list = new ArrayList<>(Arrays.asList("A", "B", "C"));

try {
    for (String item : list) {           // Uses iterator internally
        System.out.println(item);
        if (item.equals("B")) {
            list.add("D");               // Structural modification!
        }
    }
} catch (ConcurrentModificationException e) {
    System.out.println("ConcurrentModificationException thrown!");
    // Detected via modCount check inside iterator.next()
}

// CORRECT way to remove during iteration with fail-fast:
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("B")) {
        it.remove();                     // Safe: uses iterator's own remove
    }
}

// ── FAIL-SAFE: CopyOnWriteArrayList ───────────────────────────────────────
List<String> cowList = new CopyOnWriteArrayList<>(Arrays.asList("A", "B", "C"));

for (String item : cowList) {            // Iterates over a SNAPSHOT (copy)
    System.out.println(item);            // A, B, C
    cowList.add("D");                    // Adds to LIVE list, not the snapshot
}  // No exception! But "D" is NOT printed (not in snapshot)
System.out.println(cowList);             // [A, B, C, D, D, D] (added 3 times)

// ── FAIL-SAFE: ConcurrentHashMap ──────────────────────────────────────────
ConcurrentHashMap<String, Integer> concMap = new ConcurrentHashMap<>();
concMap.put("A", 1);
concMap.put("B", 2);

for (Map.Entry<String, Integer> entry : concMap.entrySet()) {
    System.out.println(entry.getKey() + "=" + entry.getValue());
    concMap.put("C", 3);               // Safe — weakly consistent iterator
}  // May or may not see "C" during iteration (weakly consistent)

Key Points:

  • Fail-fast detects concurrent modification via an internal modCount counter. Any structural change increments modCount; if the iterator detects a mismatch, it throws immediately
  • Fail-safe iterators are weakly consistent — they are safe but may not reflect the most recent state of the collection
  • CopyOnWriteArrayList is ideal for read-heavy, write-rare scenarios (e.g., event listener lists)
  • The safest way to modify a fail-fast collection while iterating is using Iterator.remove() or list.removeIf() (Java 8+)

4. What is Aspect-Oriented Programming (AOP)? Explain its concepts.

AOP: Programming paradigm that allows separating cross-cutting concerns (logging, security, transactions) from business logic.

Key Concepts:

  • Aspect: Module encapsulating cross-cutting concern (e.g., LoggingAspect)
  • Join Point: Point in program execution (method call, exception)
  • Pointcut: Expression selecting join points
  • Advice: Action taken at join point
  • Weaving: Linking aspects with application code 
@Aspect
@Component
public class LoggingAspect {

    // Pointcut expression
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() {}

    // Before advice
    @Before("serviceMethods()")
    public void logBefore(JoinPoint joinPoint) {
        System.out.println("Calling: " + joinPoint.getSignature().getName());
    }

    // After advice
    @After("serviceMethods()")
    public void logAfter(JoinPoint joinPoint) {
        System.out.println("Completed: " + joinPoint.getSignature().getName());
    }

    // Around advice
    @Around("serviceMethods()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = joinPoint.proceed();  // Execute method
        long time = System.currentTimeMillis() - start;
        System.out.println("Execution time: " + time + "ms");
        return result;
    }

    // After returning
    @AfterReturning(pointcut = "serviceMethods()", returning = "result")
    public void logResult(JoinPoint joinPoint, Object result) {
        System.out.println("Result: " + result);
    }

    // After throwing
    @AfterThrowing(pointcut = "serviceMethods()", throwing = "ex")
    public void logException(JoinPoint joinPoint, Exception ex) {
        System.out.println("Exception: " + ex.getMessage());
    }
}

12. What is @Transactional annotation?

@Transactional: Declares transaction boundaries declaratively. Spring handles begin, commit, rollback. 

@Service
public class AccountService {

    @Transactional
    public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId).get();
        Account to = accountRepository.findById(toId).get();

        from.debit(amount);
        to.credit(amount);

        accountRepository.save(from);
        accountRepository.save(to);
        // If any exception occurs, entire transaction rolls back
    }
}

16. Describe the Spring container and Spring bean lifecycle.

The Spring IoC Container is the core of the Spring Framework. It creates, configures, wires, and manages the lifecycle of application objects (called beans). Spring provides two container types: BeanFactory (lightweight) and ApplicationContext (feature-rich, used in practice). Every bean managed by the container passes through a well-defined lifecycle from instantiation to destruction.

Spring Container Overview:

                        ┌───────────────────────────────────────────────┐
                        │        Spring IoC Container                │
                        │  ┌────────────────────────────────────────────┐  │
                        │  │  Bean Definitions (config/annotations) │  │
                        │  └────────────────────────────────────────────┘  │
                        │        ↓ reads                           │
                        │  ┌────────────────────────────────────────────┐  │
                        │  │     Managed Bean Objects (singletons) │  │
                        │  └────────────────────────────────────────────┘  │
                        └───────────────────────────────────────────────┘

Complete Bean Lifecycle Phases:

 Step 1   →  Instantiate Bean        (new MyService())
 Step 2   →  Populate Properties     (inject @Autowired dependencies)

17. Describe the role of annotations in Java and how type annotations are used.

Annotations are a form of metadata that provide information about the program to the compiler, JVM, or frameworks — without changing the program's logic. They start with @ and can be applied to classes, methods, fields, parameters, and even type uses (Java 8+). Annotations replace older XML configuration and enable declarative programming.

Built-in Java Annotations:

// ── Compiler Annotations ───────────────────────────────────────────────────
@Override             // Tells compiler: this method overrides a parent method
@Deprecated           // Marks element as outdated; compiler warns on usage
@SuppressWarnings     // Tells compiler to suppress specific warnings
@FunctionalInterface  // Marks interface as having exactly one abstract method
@SafeVarargs          // Suppresses unchecked varargs warnings

// ── Meta-Annotations (annotations that annotate annotations) ─────────────
@Target      // Specifies where annotation can be applied
@Retention   // Specifies how long annotation is retained (SOURCE, CLASS, RUNTIME)
@Documented  // Include annotation in Javadoc
@Inherited   // Annotation is inherited by subclasses
@Repeatable  // Annotation can be used multiple times on same element

// ── Examples of built-in annotations in use ────────────────────────────

    public void display() { System.out.println("Parent"); }
}

    @Override                          // Compiler checks overriding is correct

Creating a Custom Annotation:

import java.lang.annotation.*;

// ── Define a custom annotation ──────────────────────────────────────────────
@Target(ElementType.METHOD)           // Only applicable to methods
@Retention(RetentionPolicy.RUNTIME)   // Visible at runtime via reflection
@Documented
public @interface AuditLog {
    String action() default "EXECUTE"; // Element with default value
    String module();                   // Required element (no default)
    boolean logResult() default false;
}

// ── Use the custom annotation ────────────────────────────────────────────────
public class UserService {

    @AuditLog(action = "CREATE", module = "Users", logResult = true)
    public User createUser(String name) {
        return new User(name);
    }

    @AuditLog(module = "Users")  // action defaults to "EXECUTE"
    public void deleteUser(Long id) {
        // delete logic
    }
}

// ── Read annotation at runtime using Reflection ─────────────────────────────
import java.lang.reflect.Method;

Class<?> clazz = UserService.class;
for (Method method : clazz.getDeclaredMethods()) {
    if (method.isAnnotationPresent(AuditLog.class)) {
        AuditLog log = method.getAnnotation(AuditLog.class);
        System.out.println("Method: " + method.getName());
        System.out.println("  action:    " + log.action());
        System.out.println("  module:    " + log.module());
        System.out.println("  logResult: " + log.logResult());
    }
}
// Output:
// Method: createUser
//   action:    CREATE
//   module:    Users
//   logResult: true

18. Explain the concept of Dependency Injection and its types in Spring.

Dependency Injection (DI) is a design pattern and a core principle of the Spring Framework that implements Inversion of Control (IoC). Instead of a class creating its own dependencies (tight coupling), the Spring IoC container injects them from outside (loose coupling). This makes code more modular, testable, and maintainable.

The Problem DI Solves:

// WITHOUT DI — tight coupling (BAD)
class OrderService {
    private PaymentService payment;

    public OrderService() {
        this.payment = new CreditCardPaymentService(); // Hard-coded dependency
        // Cannot swap to PayPal without editing this class
        // Cannot mock PaymentService in unit tests
    }
}

// WITH DI — loose coupling (GOOD)
class OrderService {
    private PaymentService payment;

    public OrderService(PaymentService payment) { // Dependency injected
        this.payment = payment;   // Can inject CreditCard, PayPal, or a mock
    }
}

Type 1 — Constructor Injection (Recommended):

Dependencies are provided through the class constructor. Spring injects them when it creates the bean. This is the recommended approach because dependencies are final, mandatory, and clearly visible. 

@Service
public class OrderService {
    private final PaymentService paymentService;  // final = immutable
    private final InventoryService inventoryService;

    // @Autowired is optional when there is only one constructor (Spring 4.3+)
    @Autowired
    public OrderService(PaymentService paymentService,
                        InventoryService inventoryService) {
        this.paymentService   = paymentService;
        this.inventoryService = inventoryService;
    }

    public void placeOrder(Order order) {
        inventoryService.reserve(order);
        paymentService.charge(order);
    }
}
// Advantages:
// ✔ Dependencies are final (immutable, thread-safe)
// ✔ Required dependencies are obvious from constructor signature
// ✔ Easy to unit test — just pass mocks via constructor
// ✔ Prevents circular dependencies at startup

Type 2 — Setter Injection:

Dependencies are injected via public setter methods after the object is constructed. Use for optional dependencies or when the dependency can be changed after construction. 

@Service
public class NotificationService {
    private EmailService emailService;     // Optional dependency
    private SMSService smsService;         // Optional dependency

    @Autowired
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }

    @Autowired(required = false)           // truly optional
    public void setSmsService(SMSService smsService) {
        this.smsService = smsService;
    }

    public void notify(String message) {
        if (emailService != null) emailService.send(message);
        if (smsService   != null) smsService.send(message);
    }
}
// Advantages:
// ✔ Supports optional dependencies with required = false
// ✔ Allows re-configuration after bean creation
// Disadvantage:
// ✘ Dependencies are not final (can be null or changed unexpectedly)

Type 3 — Field Injection (Avoid in Production):

Spring injects the dependency directly into the field using reflection. Concise but not recommended because it hides dependencies, prevents final, and makes testing harder without a Spring context. 

@Service
public class ReportService {
    @Autowired
    private UserRepository userRepository;   // Injected via reflection

    @Autowired
    private OrderRepository orderRepository;

    public Report generateReport() {
        // use repositories
    }
}
// Disadvantages:
// ✘ Cannot make fields final
// ✘ Hidden dependencies — not visible in constructor
// ✘ Requires Spring context to test (cannot use plain new ReportService())
// ✘ Encourages too many dependencies (code smell)

Comparison Table:

Aspect Constructor Injection Setter Injection Field Injection
Recommended? ✔ Yes (default choice) Sometimes ✘ Avoid
Dependencies final? Yes No No
Mandatory deps? Enforced at startup Not enforced Not enforced
Unit testing Easy (new MyClass(mock)) Possible (setXxx(mock)) Hard (needs reflection)
Circular dependency Detected at startup Allowed (resolved lazily) Allowed

Key Points:

  • IoC is the principle (container controls object creation); DI is the mechanism (container injects dependencies)
  • Spring supports all three types but officially recommends Constructor Injection for required dependencies
  • @Autowired tells Spring where to inject; @Qualifier disambiguates when multiple beans of the same type exist
  • DI enables easy unit testing by allowing mock objects to be substituted for real dependencies

package com.example.demo.service;

import com.example.demo.model.Student;
import org.springframework.stereotype.Service;
import java.util.*;

@Service    // Marks as a Spring-managed service bean
public class StudentService {

    // In-memory list — simulates a database
    private final List<Student> students = new ArrayList<>();
    private long nextId = 1;

    // Pre-load some data
    public StudentService() {
        students.add(new Student(nextId++, "Alice", "CSE", 2));
        students.add(new Student(nextId++, "Bob",   "ECE", 3));
    }

    // READ ALL — returns every student
    public List<Student> getAllStudents() {
        return students;
    }

    // READ ONE — find by id, return Optional
    public Optional<Student> getStudentById(Long id) {
        return students.stream()
                       .filter(s -> s.getId().equals(id))
                       .findFirst();
    }

    // CREATE — assigns auto-incremented id and adds to list
    public Student addStudent(Student student) {
        student.setId(nextId++);
        students.add(student);
        return student;
    }

    // UPDATE — find by id and overwrite fields
    public Optional<Student> updateStudent(Long id, Student updated) {
        Optional<Student> existing = getStudentById(id);
        existing.ifPresent(s -> {
            s.setName(updated.getName());
            s.setBranch(updated.getBranch());
            s.setYear(updated.getYear());
        });
        return existing;
    }

    // DELETE — remove by id, return true if removed
    public boolean deleteStudent(Long id) {
        return students.removeIf(s -> s.getId().equals(id));
    }
}

package com.example.demo.controller;

import com.example.demo.model.Student;
import com.example.demo.service.StudentService;
import org.springframework.http.*;
import org.springframework.web.bind.annotation.*;
import java.util.List;

@RestController                    // @Controller + @ResponseBody
@RequestMapping("/api/students")   // Base URL for all endpoints
public class StudentController {

    private final StudentService service;

    public StudentController(StudentService service) {  // Constructor DI
        this.service = service;
    }

    // ─ GET /api/students — Retrieve ALL students
    @GetMapping
    public ResponseEntity<List<Student>> getAll() {
        return ResponseEntity.ok(service.getAllStudents());   // 200 OK
    }

    // ─ GET /api/students/{id} — Retrieve ONE student
    @GetMapping("/{id}")
    public ResponseEntity<Student> getById(@PathVariable Long id) {
        return service.getStudentById(id)
                      .map(ResponseEntity::ok)              // 200 OK
                      .orElse(ResponseEntity.notFound().build());  // 404
    }

    // ─ POST /api/students — CREATE a new student
    @PostMapping
    public ResponseEntity<Student> create(@RequestBody Student student) {
        Student created = service.addStudent(student);
        return ResponseEntity.status(HttpStatus.CREATED).body(created); // 201
    }

    // ─ PUT /api/students/{id} — UPDATE an existing student
    @PutMapping("/{id}")
    public ResponseEntity<Student> update(
            @PathVariable Long id,
            @RequestBody Student updatedStudent) {
        return service.updateStudent(id, updatedStudent)
                      .map(ResponseEntity::ok)              // 200 OK
                      .orElse(ResponseEntity.notFound().build());  // 404
    }

    // ─ DELETE /api/students/{id} — DELETE a student
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        boolean removed = service.deleteStudent(id);
        return removed
            ? ResponseEntity.noContent().build()          // 204 No Content
            : ResponseEntity.notFound().build();          // 404 Not Found
    }
}

20. Describe the structure of a basic Spring Boot application. What are the advantages of using Spring Boot over traditional web frameworks?

A Spring Boot application follows a layered architecture with a specific project structure generated by Spring Initializr. It packages everything (including an embedded server) into a single executable JAR. Traditional web frameworks (like Servlet/JSP + Spring MVC without Boot) require external application servers, verbose XML configuration, and manual dependency management.

Standard Spring Boot Project Structure: 

my-app/
├── pom.xml                              ← Maven build file (dependencies)
└── src/
    ├── main/
    │   ├── java/com/example/myapp/
    │   │   ├── MyAppApplication.java    ← @SpringBootApplication entry point
    │   │   ├── controller/              ← @RestController — HTTP layer
    │   │   │   └── StudentController.java
    │   │   ├── service/                 ← @Service — business logic
    │   │   │   └── StudentService.java
    │   │   ├── repository/              ← @Repository — data access (JPA)
    │   │   │   └── StudentRepository.java
    │   │   └── model/                   ← @Entity / POJO — data model
    │   │       └── Student.java
    │   └── resources/
    │       ├── application.properties    ← Configuration (port, DB URL, etc.)
    │       └── static/                   ← Static files (HTML, CSS, JS)

Advantages of Spring Boot (Summary):

  • Auto-configuration — detects libraries on classpath and configures beans automatically (e.g., if H2 jar is present, configures an in-memory DataSource)
  • Embedded server — Tomcat/Jetty/Undertow bundled inside the JAR; no separate server installation needed
  • Starter dependencies — spring-boot-starter-web pulls in all web dependencies with compatible versions in one line
  • Production-ready — Spring Boot Actuator provides /actuator/health, /actuator/metrics, /actuator/info out of the box
  • Minimal boilerplate — no web.xml, no XML context files; a single class with main() is enough to run a full web app
  • Opinionated defaults — sensible defaults for most settings, easily overridden in application.properties
  • Microservices-ready — pairs with Spring Cloud for service discovery, circuit breakers, and API gateways
